Payment Reconciliation Service

A backend service that ingests payment lifecycle events, tracks transaction state, and surfaces settlement discrepancies. Built in phases — each phase is independently testable before moving forward.

FrameworkFastAPI (Python)
DatabaseSupabase PostgreSQL (managed)
ORMSQLAlchemy 2.0

What We're Building

Payment systems generate events as money moves through them. A single payment produces a chain of events — it starts, gets processed, then either succeeds and gets settled, or fails. The service I've designed is a backend that:

Core Capabilities
  • Receives and safely stores payment lifecycle events
    Handles duplicates gracefully — the same event arriving twice won't corrupt data
  • Maintains the current state of every transaction
    One row per transaction that gets updated as events arrive in order
  • Provides queryable APIs for operations teams
    Filter by merchant, status, and date with full pagination and sorting
  • Surfaces reconciliation discrepancies automatically
    Finds payments processed but never settled, failed payments with settlement records, and stuck initiations

// Reconciliation: The Core Problem

Settlement operates as a separate system from payment processing. These two systems should always agree — but sometimes they don't. When they diverge, those gaps become discrepancies:

Type What happened Why it matters
processed → not settled Payment approved, merchant never received money Merchant loss — needs urgent investigation
failed → settled Payment rejected but settlement exists Possible double credit — financial risk
stuck at initiated Payment started, no further events ever arrived Payment processor may have dropped the request

Key design principle: Each file is architected with exactly one responsibility. Business logic stays cleanly separated across layers — database access lives in routers using raw SQL queries, while models and schemas remain free of business logic. All validation happens at the Pydantic layer before any database operations ever touch the tables.

// Why This Structure

  • app/database.py — Centralized DB configuration. Change one file to switch databases or adjust connection pooling.
  • app/models.py — ORM layer mirrors schema.sql exactly. ENUM definitions live here so Pydantic and DB stay in sync.
  • app/schemas.py — Request/response contracts. All validation rules (amount > 0, event_type in enum, etc) enforced before database touch.
  • app/routers/ — Each endpoint is its own file. Transactions API doesn't know about events router logic.
  • db/ — SQL is the source of truth. Views abstract complex reconciliation queries.

02Phase

Database Schema

This phase establishes the complete database structure: 3 core tables, enums, indexes, and triggers. The SQL is applied to Supabase, and the structure is validated before any API code is written.

// Schema Design

Table: merchants
Column Type Why this type
merchant_id VARCHAR(50) PK Short identifier like "merchant_1". PK = unique + indexed automatically.
merchant_name VARCHAR(255) Human-readable name. NOT NULL — every merchant must have a name.
created_at TIMESTAMPTZ TZ = timezone-aware. Always store UTC for financial records.
updated_at TIMESTAMPTZ Auto-updated by a DB trigger — not Python code.
Table: transactions
Column Type Why this type
transaction_id VARCHAR(100) PK UUID from upstream system. VARCHAR so we accept any format.
merchant_id VARCHAR(50) FK Foreign key to merchants. DB rejects unknown merchant IDs.
amount NUMERIC(12,2) NOT Float. Exact decimal for money. Stores up to 9,999,999,999.99
currency VARCHAR(10) Default 'INR'. Allows multi-currency later.
status transaction_status_enum DB-level ENUM. Rejects invalid statuses before app code runs.
created_at TIMESTAMPTZ When this transaction first appeared in our system.
updated_at TIMESTAMPTZ Auto-updated by trigger on every status change.
INDEXES → merchant_id status created_at DESC (merchant_id, status) composite
Table: events
Column Type Why this type
event_id VARCHAR(100) PK PRIMARY KEY enforces idempotency — duplicate INSERT fails at DB level.
transaction_id VARCHAR(100) FK Links this event to its transaction. FK ensures transaction must exist.
event_type event_type_enum DB ENUM — only valid event types accepted.
timestamp TIMESTAMPTZ When the event HAPPENED in the source system. From the payload.
created_at TIMESTAMPTZ When WE received it. Different from timestamp during retries/delays.
INDEXES → transaction_id (event_type, transaction_id) composite

03Phase

Event Ingestion API

The POST /events endpoint is where event ingestion happens. This is the most critical endpoint because it implements both idempotency and state machine correctness.

// What This Endpoint Does (step by step)

POST /events — Decision Flow
Receive event payload
↓ Pydantic validates shape + types
↓ Is event_id already in events table?
YES → Return 200 "duplicate ignored". Stop.
↓ NO — continue
↓ Does merchant exist?
NO → Create merchant row first
↓ Does transaction exist?
NO → Create transaction with status = event_type
YES → Is this a valid state transition?
INVALID → Return 200 "invalid transition". Stop.
VALID → Update transaction.status
↓ Append event row to events table
↓ Commit everything atomically
✓ Return 200 "event recorded"

// What Gets Built

  • app/database.py — Handles DB connection pooling for PostgreSQL
  • app/models.py — SQLAlchemy ORM classes that mirror schema.sql exactly
  • app/schemas.py — Pydantic EventIn model with field validators
  • app/routers/events.py — The POST /events endpoint with full event ingestion logic
  • main.py — Updated to register the events router

// Validation Rules (enforced by Pydantic)

Field Rule Reject if...
event_type Must be in allowed set Any string not in {payment_initiated, payment_processed, payment_failed, settled}
amount Must be positive Zero, negative, or missing
event_id Must be non-empty string Empty string or missing
timestamp Must be valid datetime Unparseable date string
transaction_id Must be non-empty string Empty or missing

05Phase

Reconciliation APIs

The reconciliation layer consists of two endpoints that expose discrepancy detection. All the complex logic is implemented in SQL views; Python simply queries those views and formats responses.

// GET /reconciliation/summary

Aggregated counts and totals. Groups by merchant or status. Uses SQL GROUP BY + SUM + COUNT — not Python loops.

Example Response — group_by=merchant
[
  { "group": "merchant_1", "total_transactions": 412, "total_amount": 2840192.50 },
  { "group": "merchant_2", "total_transactions": 389, "total_amount": 1920441.00 }
]

// GET /reconciliation/discrepancies

Returns transactions where something is wrong. Implemented as 3 SQL views merged with UNION ALL.

View SQL Logic
v_processed_not_settled status = 'payment_processed' AND no 'settled' event exists for this transaction
v_failed_but_settled status = 'payment_failed' AND a 'settled' event exists for this transaction
v_stuck_initiated status = 'payment_initiated' AND no event beyond initiation exists
Example Response
{
  "total_discrepancies": 47,
  "discrepancies": [
    {
      "transaction_id":  "abc-123",
      "merchant_id":     "merchant_2",
      "amount":          15248.29,
      "status":          "payment_processed",
      "discrepancy_type": "processed but never settled"
    }
  ]
}

Assumptions & Design Decisions

// Data Assumptions

  • Events arrive out-of-order
    The system must handle events arriving in any sequence. Priority system ensures state consistency regardless of order.
  • Event IDs are globally unique
    No UUID collisions. Idempotency relies on event_id being a reliable primary key.
  • Merchants must be created before transactions
    The API auto-creates merchants on first event. No manual merchant registration needed.
  • Amounts are always positive
    Negative amounts or refunds are not in the MVP. Pydantic validates amount > 0.
  • Amount is consistent across transaction
    For all events belonging to a transaction the amount stays same.
  • Currency is ISO 4217 compliant (e.g., "INR", "USD")
    Default is INR. Multi-currency is supported but not enforced.
  • Timestamps in events are UTC
    TIMESTAMPTZ stores timezone info. Assumes all incoming timestamps are timezone-aware or UTC.

// Architectural Assumptions

  • State transitions are immutable
    Once a transaction reaches 'settled' or 'payment_failed', no further state changes allowed. This is enforced by priority system (higher priority blocks lower priority).
  • Events are immutable
    Events table is append-only. No UPDATE or DELETE on events. Full audit trail is preserved forever.
  • Duplicate detection is critical
    Same event arriving twice (same event_id) must be silently ignored. Database enforces at PRIMARY KEY level.
  • Consistency over availability
    If DB connection fails, the API returns 5xx rather than serving stale data. PostgreSQL transactions ensure ACID compliance.
  • No batch operations in API
    MVP only accepts one event per POST /events call. Bulk loading happens via seed.py, not the API.
  • Timestamps are source-of-truth
    Event ordering is based on the 'timestamp' field (when event happened), not 'created_at' (when we received it). This allows recovery from delays.

// Operational Assumptions

  • Database exists and schema is initialized
    schema.sql must be run before the app starts. No auto-migrations in the MVP.
  • Supabase credentials are stored in .env
    DATABASE_URL is the only config needed. No multi-tenancy or complex secrets.
  • Rate limiting is not implemented
    Assumes trusted sources. No API key authentication or request throttling in MVP.
  • No transaction rollback for duplicates
    Duplicate events are caught at PRIMARY KEY level after partial inserts. IntegrityError is caught and logged.
  • Reconciliation views are manually maintained
    No automated triggers to update views. Views are recalculated on every query.

// Security & Performance Assumptions

  • No encryption of data at rest
    MVP assumes Supabase SSL/TLS for transit. App-level encryption not implemented.
  • Connection pooling prevents DB exhaustion
    Pool size = 3, max_overflow = 2. Tuned for small workloads. Not tested under high concurrency.
  • Indexes are sufficient for current data volume
    10,000 events is small. B-tree indexes on merchant_id, status, created_at should be fast.
  • No query optimization for large scans
    Discrepancies endpoint UNIONs three views. Performance assumes views are materialized efficiently by PostgreSQL.
